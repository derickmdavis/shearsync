# Cover Photo Debug Guide

This document explains how the ShearSync API stores and returns the booking-page cover image so the web app can debug missing images correctly.

## Source Of Truth

The booking-page cover image is stored on the `stylists` record as:

- `cover_photo_url`

This is a plain URL string, not a storage path and not a signed URL generated by the backend.

## What The Backend Expects

When saving booking settings, the backend accepts:

- `cover_photo_url: string`
- or `cover_photo_url: ""` to clear it

Validation is in [src/validators/settingsValidators.ts](/Users/derick/shearsync-api/src/validators/settingsValidators.ts:11):

- `cover_photo_url` must be a valid absolute URL when present
- empty string is allowed for clearing

Normalization is in [src/services/stylistsService.ts](/Users/derick/shearsync-api/src/services/stylistsService.ts:9):

- `""` is normalized away
- real URLs are stored as-is

Important:

- the backend does not transform the URL
- the backend does not sign the URL
- the backend does not prepend a CDN or asset base URL
- the backend does not fetch the image

If the URL in the database is wrong, expired, private, or blocked by CORS/storage rules, the frontend will not be able to render it.

## Endpoints The Web App Should Use

### 1. Internal settings/editor view

Use:

- `GET /api/settings/booking`

This returns the authenticated user’s stylist/booking settings row, including `cover_photo_url`.

Controller:

- [src/controllers/settingsController.ts](/Users/derick/shearsync-api/src/controllers/settingsController.ts:18)

Service:

- [src/services/stylistsService.ts](/Users/derick/shearsync-api/src/services/stylistsService.ts:25)

What the web app should expect:

```json
{
  "data": {
    "id": "stylist_uuid",
    "user_id": "user_uuid",
    "slug": "maya-johnson",
    "display_name": "Maya Johnson",
    "bio": "Hair color specialist",
    "cover_photo_url": "https://your-public-image-url.example.com/cover.jpg",
    "booking_enabled": true
  }
}
```

Notes:

- This is the correct endpoint for settings/admin screens.
- If `cover_photo_url` is missing here, the DB row may not contain it, or the frontend may be reading the wrong endpoint.

### 2. Public booking page

Use:

- `GET /api/public/stylists/:slug`

Controller:

- [src/controllers/publicController.ts](/Users/derick/shearsync-api/src/controllers/publicController.ts:9)

Service:

- [src/services/stylistsService.ts](/Users/derick/shearsync-api/src/services/stylistsService.ts:37)

Response type:

- [src/types/api.ts](/Users/derick/shearsync-api/src/types/api.ts:195)

What the web app should expect:

```json
{
  "data": {
    "id": "stylist_uuid",
    "slug": "maya-johnson",
    "display_name": "Maya Johnson",
    "bio": "Hair color specialist",
    "cover_photo_url": "https://your-public-image-url.example.com/cover.jpg",
    "booking_enabled": true,
    "business_name": "Maya Johnson Hair",
    "phone_number": "555-555-5555",
    "timezone": "America/Denver"
  }
}
```

Important:

- The public booking page should read the cover image from `data.cover_photo_url`.
- It should not expect `coverPhotoUrl`.
- It should not expect the image inside `profile`, `hero`, or `settings`.
- It should not expect the image from `GET /api/profile/overview`.

## Exact Field Name

The API returns:

- `cover_photo_url`

If the web app expects:

- `coverPhotoUrl`

then it must map the server response explicitly.

Example:

```ts
const coverPhotoUrl = response.data.cover_photo_url ?? null;
```

## How The URL Is Returned

The public profile service returns the DB value directly:

```ts
cover_photo_url: (stylist.cover_photo_url as string | null | undefined) ?? null
```

Source:

- [src/services/stylistsService.ts](/Users/derick/shearsync-api/src/services/stylistsService.ts:41)

That means:

- if the DB contains a real URL string, the API returns that string
- if the DB contains `null`, the API returns `null`
- if the DB contains `""`, it should usually have been normalized away before storage

## Common Frontend Debugging Checks

### 1. Confirm the frontend is calling the correct endpoint

For a public booking page:

- `GET /api/public/stylists/:slug`

For an internal settings page:

- `GET /api/settings/booking`

### 2. Confirm the frontend is reading the correct field

Use:

- `response.data.cover_photo_url`

Not:

- `response.cover_photo_url`
- `response.data.coverPhotoUrl`
- `response.profile.cover_photo_url`

### 3. Confirm the slug is correct

If the web app loads the wrong stylist slug, it may get a different stylist record with no cover image.

### 4. Confirm the API payload really contains the URL

Inspect the raw network response in the browser devtools.

If the API response already contains:

- `"cover_photo_url": "https://..."`

then the backend contract is working and the issue is in frontend mapping or rendering.

### 5. Confirm the image URL is publicly reachable

Open the exact returned URL directly in the browser.

If it does not load, the likely issue is:

- private storage object
- expired signed URL
- incorrect path
- storage bucket permissions
- removed asset

### 6. Confirm the web app is not filtering it out

Look for frontend logic like:

- only show image when `booking_enabled === true`
- fallback image overrides
- conditional rendering tied to wrong field names
- sanitizer code dropping snake_case fields

## What The Backend Does Not Do

The backend currently does not:

- upload the image
- verify the remote image exists
- convert a storage file path into a public URL
- generate signed image URLs
- return a transformed image object

So if your database stores a raw storage path like:

- `client-photos/user-id/image.jpg`

instead of a real public URL like:

- `https://...`

the web app will not be able to render it directly.

## Recommended Web App Usage

For the public booking page:

1. Request `GET /api/public/stylists/:slug`
2. Read `data.cover_photo_url`
3. If non-null, render it directly in `<img src={coverPhotoUrl}>` or as a CSS background image
4. If null, render the default fallback image/state

## Quick Diagnosis Matrix

If DB has URL but page shows no image:

- API response missing URL:
  likely wrong endpoint, wrong slug, or stale backend data path

- API response includes URL but UI still blank:
  likely frontend field-mapping bug

- API response includes URL and frontend maps it, but browser image request fails:
  likely invalid/private/unreachable asset URL

- Internal settings page shows image but public booking page does not:
  likely public page is not calling `GET /api/public/stylists/:slug` correctly
